## 导语

大部分人用 AI 写代码做研究，就是丢个需求进去，然后一轮轮改 bug。写出来的代码的一致性，鲁棒性，以及工程性都不够好。

但其实 AI（Claude，Gemini，Cursor 之类）都支持一种叫 **System Prompt** 的机制——你可以提前写一份"规范文档"放在项目根目录，AI 每次帮你写代码之前都会先读这份文件。

相当于你给 AI 写了一本员工手册。

今天我就把我打磨了许久的“AI 工作手册”的核心模块拆解一下，希望对同样用 AI 搞研究、写策略的朋友有帮助。

## 一、Python 编码规范

**指令：**

```rust
- Python 3.10+，强制类型标注
- 严格遵守PEP 8，用ruff做格式化和检查
- 所有公开接口必须有类型提示
- Google风格的文档字符串，核心函数要标注时间/空间复杂度
- 每个文件必须有 `if __name__ == "__main__"` 入口，带真实的示例数据
- 性能关键路径用 `time.perf_counter()` 计时
- 注释用中文，变量名和日志用英文
```

## 

 **为什么这么写？**

这些规则解决的是 AI 写代码"能跑但不规范"的问题。没有这条约束的时候，AI 经常：

1. 类型标注随缘写，你后面接手根本不知道传什么进去
2. 不写 **MRE**（最小可复现示例），你收到代码不知道怎么单独测试
3. 变量名用中文拼音，后面 grep 搜日志的时候搜不到
4. 从来不考虑性能，写完了你自己还得 benchmark

加了这些规则之后，AI 每次产出的代码至少是"有类型、有示例、可测试"的状态，省了大量返工时间。

## 二、数据处理和量化通用规则

### 2.1 数据完整性

**指令：**

```rust
- 所有数据处理必须显式处理缺失值、Inf、除零、空DataFrame
- 严禁未来函数：回测中不得使用历史上不可获得的数据，复权函数必须统一管理
- 成交映射：委托执行和持仓估值必须映射回原始未复权价格
```

 ##
 
 **为什么这么写？**

AI 最容易犯的错就是**未来函数**，在回测里看起来收益炸裂，但根本不可能实现。

还有复权的坑。很多人让 AI 直接用前复权价格做回测，结果算出来的收益率是失真的，因为前复权价格会随分红送股不断调整。正确做法是用复权价格计算信号，但成交撮合和盈亏计算要映射回原始价格。

> 把这些规则写进手册之后，AI 在写策略代码时会主动加上 look-ahead bias 的检查注释，也会自动把复权逻辑和交易逻辑分开，出错概率降低了很多。

##
### 2.2 性能优先级

**指令：**

```rust
- 向量化优先：优先使用Pandas/NumPy/Polars的内置向量化操作
- 禁止逐行迭代：禁止在大DataFrame上使用`.iterrows()`等逐行循环，实在需要循环就用numba
- 内存评估：处理大规模因子数据集时必须评估峰值内存
```

##

**为什么这么写？**

AI 默认写 Python 代码的思路是"先跑通再说"，经常写出 for 循环遍历十万行数据的代码。在量化场景下，你可能要处理几千只股票、几年的日频甚至分钟频数据，一个不小心就是几个 G 的 DataFrame 在内存里。

加了这条规则后，AI 会优先用向量化的方式实现——比如用 `df.groupby().transform()` 代替循环，用 Polars 的 LazyFrame 做惰性求值，这些都能让性能提升几个数量级。

##

### 2.3 双模式存储

**指令：**

```rust
- Parquet：本地缓存，用于轻量级离线研究和简单时序分析
- ClickHouse：列式存储，用于高频/海量行情查询，是实盘和大规模回测的核心数据源
```

**为什么这么写？**

做研究的时候经常是"先随便拉点数据跑跑看"，这时候 Parquet 文件最方便，读写快、兼容好。但一旦进入正式回测或者上实盘，Parquet 就扛不住了——你需要一个真正的列式数据库来支撑海量查询。

## 三、日志和异常处理

**指令：**

```rust
- 禁止`print()`，全部替换为结构化日志
- 日志分级：DEBUG看中间数据，INFO记关键节点，WARNING记预期内的异常（如节假日缺数据），ERROR记系统中断
- 容错：网络请求、API、文件IO必须try-except包裹，主线程不能因单个任务挂掉
- 告警：关键阻塞问题要支持企微/飞书/邮件通知
```

**为什么这么写？**

结构化日志可以接入告警系统，出问题自动发消息。

## 四、架构原则

**指令：**

```rust
- 数据和逻辑分离：策略代码里不能硬编码数据获取，数据目录物理隔离
- 网关适配器模式：策略计算层和券商/数据商执行层解耦，异构接口在网关层标准化
- 参数外置：均线周期、止损比例等可变参数通过JSON/YAML注入，不能硬编码
- 双执行模式：回测引擎同时提供向量化模式（快速因子筛选）和逐Tick事件驱动模式（与实盘一致）
```

##

**为什么这么写？**

没有架构约束的话，AI 特别喜欢写"一把梭"的代码：数据获取、信号计算、下单执行全塞在一个函数里，参数直接写死。这种代码改一个均线周期都要改源码，换个券商接口要重写半个策略。

有了架构规则，AI 产出的代码天然就是分层的：数据层、策略层、执行层各管各的，参数从配置文件读，换个数据源或券商只需要改适配器。

## 五、回测和资金管理

**指令：**

```rust
- 必须加入滑点和手续费模型
- 买入信号必须验证资金是否充足（保证金比例、爆仓风险），禁止无限资金假设
- 上线流程：历史回测（含滑点手续费）→ 模拟盘 → 小资金实盘，三步验证
```

**为什么这么写？**

*"回测年化 30%，实盘亏 20%"*的原因，通常是没有经过模拟盘验证、滑点估计不足、流动性假设过于乐观。

## 六、可视化规范

**指令：**

```rust
- 默认使用交互式图表（pyecharts等支持缩放/悬停的），只有明确要静态图时才用matplotlib/seaborn
- AI应该主动搜索和推荐最佳可视化组件
- 新图表库要先验证兼容性
```

**为什么这么写？**

matplotlib 画出来的图是死的，放大缩小都不行。pyecharts 或者 TradingView 的轻量级图表在可视化方面很好用。

## 七、Git 工作流

**指令：**

```rust
- main分支绝对保护，只有完整测试+回测稳定的代码才能合入
- dev分支日常维护和集成
- 新因子/新网关走feature分支，完成后提PR
- 实验策略走私有分支，不能污染主策略库
- 提交信息用约定式格式
```

## 八、AI 协作原则

**指令：**

```rust
- 有歧义的金融公式/策略算法必须先确认，不能自己编
- 修改现有文件时遵循已有的命名规范和调用方式
- 优先小改动而非重写，保护上下游依赖
- 可以主动建议优化方案，但实质性改动必须经过确认才动手
- 先给代码，再给简短解释，明确标注约束条件
```

## 总结

> 一份手册，省几百个小时。

如果你也在用 AI 辅助做量化研究，强烈建议花时间写一份自己的规范文档。不需要一步到位，可以在日常使用中逐步积累——每次 AI 犯了一个让你恼火的错误，就把对应的规则加进去。

**用过之后你会发现：AI 不是不好用，是你没教它怎么做。**
